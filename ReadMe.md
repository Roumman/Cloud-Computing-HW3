# Задание: развёртывание Flask-приложения с использованием blue-green deployment и Continuous Delivery
### Цель:
развернуть приложение «Hello World» на Flask (с поддержкой эндпоинта /version), используя облачный реестр контейнеров и реализовать механику blue-green deployment с Continuous Delivery.

## 1. Подготовка и деплой образа (базовые шаги)
Необходимо выполнить следующие действия для деплоя приложения:

Собрать Docker-образ приложения, передав текущую версию в качестве аргумента (например, VERSION=1.0.0).
Запушить образ в Container Registry от Яндекс (сервис: ) — это обеспечит быстрый доступ к образам с виртуальных машин в облаке.
Запустить виртуальную машину (вручную, без необходимости создавать новую машину при каждом релизе).
Написать скрипт (bash или ansible), который:
заходит на виртуальную машину;
либо запускает сервис с нуля;
либо меняет версию контейнера без остановки входящего трафика (критично для blue-green deployment).
Настроить пайплайн в GitLab CI/CD, чтобы после пуша с нужным тегом:
собирался контейнер с версией тега;
образ сервиса пушился в облачный Docker Registry с тегом новой версии;
запускался скрипт из п. 4 для развёртывания новой версии сервиса.

## 2. Работа с облачным registry (2 балла)
Пушить в облако только образы, готовые к релизу.
Использовать теги коммитов (например, v123) для именования образов (формат: example.com/registry/service-name:v123).
При пуше тега ($CI_COMMIT_TAG) в GitLab:
протэгать образ соответствующим образом;
залогиниться в облачный registry и запушить образ.

## 3. Реализация blue-green deployment (3 или 5 баллов)
Задача: обновить версии сервисов с минимальными потерями трафика.

### Механизм:

Использовать роутер/балансировщик перед сервисами (или перед всеми сервисами, если есть доменные имена).
Балансировщик направляет трафик на один или несколько инстансов сервиса.
При обновлении:
Запускается новая версия сервиса рядом с текущей.
Меняется конфигурация балансировщика.
Трафик переключается на новую версию.

## 4. Continuous Delivery (3 балла)

### Задача: автоматизировать деплой новых версий сервисов по нажатию кнопки.
### Решение: 
Использовать Gitlab CI/CD и manual jobs:
- создать Job, которая не запускается автоматически, а появляется в интерфейсе как кнопка;
- запускать Job на пуш с тегом версии;
- для доступа пайплайна к серверам — положить приватный ключ в переменную окружения в CI;
- в пайплайне запускать скрипт или Ansible playbook для деплоя.

## 5. Бонус: безопасность (2 балла)
Чтобы получить бонусные баллы, нужно обеспечить безопасность работы с Container Registry:
- создать сервисный аккаунт с доступом только к Container Registry (для загрузки/выгрузки образов);
- использовать авторизованные ключи (SSH-ключи для релиза не должны использоваться где-либо ещё);
- не хранить ключи в Git-репозитории — использовать переменные окружения или защищённые файлы;
- обеспечить, чтобы сервисы принимали только локальные подключения (если всё на одной машине) или были за NAT (если сервисы на отдельных машинах);
- исключить возможность доступа к сервису в обход прокси.
